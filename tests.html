Test Guide

This site is a QA Dashboard — but its real purpose is to serve as the target application for an automated Playwright test suite. Key pages, components, and interactions have corresponding tests that validate them. This page documents each test suite: what it tests, how it works, and the Playwright APIs it demonstrates.

This project combines a small dashboard with 100+ automated checks covering smoke flows, content validation, interactions, responsive layout, API mocking, accessibility, test cases, bug reports, status reports, and project documentation.

How to run the tests

npm install
npx playwright install --with-deps
npx http-server . -p 3000 -c-1
npx playwright test

Run the suite across the configured Playwright projects. Add --project=chromium to run a single browser. Use --ui for the interactive Playwright UI mode.

01 · Smoke Tests

page.goto toHaveTitle toBeVisible
Verifies that every route loads successfully, the page title is correct, all navigation links are present and clickable, and a 404 is handled gracefully. These are the first tests to run — if any fails, the build stops.
File: tests/smoke.spec.js Tests: 4 Validates: all 4 pages
View code sample
// Each page is tested in a loop: const pages = [ { path: '/', title: /Playwright QA Lab.*Dashboard/i }, { path: '/scenarios.html', title: /Playwright QA Lab.*Scenarios/i }, { path: '/history.html', title: /Playwright QA Lab.*History/i }, { path: '/endpoints.html', title: /Playwright QA Lab.*Endpoints/i }, ]; for (const { path, title } of pages) { await page.goto(path); await expect(page).toHaveTitle(title); } // 404 test: const response = await page.goto('/nonexistent.html'); expect(response.status()).toBe(404);

02 · Content Validation

page.locator toHaveCount not.toBeEmpty toContainText
Checks that all UI elements render with correct data. Validates metric cards (labels, values, subtext), table headers and row counts, scenario card titles, and endpoint list items. This ensures the JavaScript-rendered content is correct.
File: tests/content.spec.js Tests: 10 Validates: Dashboard metrics, tables, scenario cards, endpoints
View code sample
// Verify 4 metric cards exist: const cards = page.locator('.metric-card'); await expect(cards).toHaveCount(4); // Each card has label + value + subtext: for (let i = 0; i < count; i++) { await expect(card.locator('.metric-label')).not.toBeEmpty(); await expect(card.locator('.metric-value')).not.toBeEmpty(); } // Table headers are correct: await expect(headers).toContainText(['Run', 'Date', 'Status']);

03 · Interaction Tests

fill selectOption toContainText input event
Tests real user interactions — typing in the search box, selecting dropdown filters, combining search + filters, and verifying the empty state when no results match. Also validates that clicking navbar links navigates to the correct pages.
File: tests/interaction.spec.js Tests: 7 Validates: Search, status/suite filters, combined filters, empty state, navigation
View code sample
// Type in search and verify results match: await page.locator('#search-input').fill('Login'); const cards = page.locator('.scenario-card'); for (let i = 0; i < count; i++) { await expect(cards.nth(i).locator('h3')).toContainText(/Login/i); } // Select filter: await page.locator('#status-filter').selectOption('failed'); // Empty state: await page.locator('#search-input').fill('zzzzzthisdoesnotexist'); await expect(page.locator('.scenario-grid')).toContainText(/No scenarios match/i);

04 · Responsive Tests

setViewportSize scrollWidth check evaluate
Tests the layout at 5 viewport sizes (375px mobile to 1920px desktop). For each size, checks there is no horizontal overflow, then verifies grid layout adapts correctly (metrics grid columns, single-column on mobile).
File: tests/responsive.spec.js Tests: 22 viewport checks Validates: 5 viewports × 4 pages + 2 layout checks
View code sample
// Test 5 viewports across all 4 pages: const viewports = [ { name: 'mobile', width: 375, height: 812 }, { name: 'tablet', width: 768, height: 1024 }, { name: 'desktop', width: 1920, height: 1080 }, ]; await page.setViewportSize({ width: 375, height: 812 }); // No horizontal overflow: const overflowWidth = await page.evaluate(() => document.documentElement.scrollWidth - document.documentElement.clientWidth ); expect(overflowWidth).toBe(0);

05 · API Mocking

page.route route.fulfill route.abort response status
Demonstrates Playwright's network interception — mocking API responses, simulating server errors (500), network delays, and connection failures. This proves I can test how an application behaves under different backend conditions without needing a real server.
File: tests/api-mocking.spec.js Tests: 5 Validates: Mock success, error, delay, abort, request interception
View code sample
// Mock a successful API response: await page.route('**/api/v2/health', async route => { await route.fulfill({ status: 200, contentType: 'application/json', body: JSON.stringify({ status: 'ok', version: '2.1.0' }), }); }); // Mock a 500 error: await page.route('**/api/v2/products', async route => { await route.fulfill({ status: 500, body: JSON.stringify({ error: 'Internal Server Error' }), }); }); // Simulate network failure: await page.route('**/api/v2/notifications', async route => { await route.abort('connectionrefused'); });

06 · Accessibility Checks

toHaveAttribute getAttribute locator count
Validates basic accessibility requirements — the page has a lang attribute, all images have alt text, form controls have aria-labels, nav links are keyboard-accessible, and headings follow a logical hierarchy (single h1).
File: tests/accessibility.spec.js Tests: 6 Validates: lang attribute, alt text, aria-labels, heading hierarchy, keyboard nav
View code sample
// Page has a lang attribute: const lang = await page.locator('html').getAttribute('lang'); expect(lang).toBeTruthy(); // All images have alt text: const images = page.locator('img'); for (let i = 0; i < count; i++) { await expect(images.nth(i)).toHaveAttribute('alt'); } // Search input has accessible label: await expect(page.locator('#search-input')) .toHaveAttribute('aria-label', /search/i);
